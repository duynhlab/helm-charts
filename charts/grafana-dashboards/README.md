# grafana-dashboards

![Version: 0.1.1](https://img.shields.io/badge/Version-0.1.1-informational?style=flat-square) ![Type: application](https://img.shields.io/badge/Type-application-informational?style=flat-square) ![AppVersion: 1.0.0](https://img.shields.io/badge/AppVersion-1.0.0-informational?style=flat-square)

Helm chart that deploys Grafana dashboards as Kubernetes ConfigMaps.
Works with Grafana sidecar to automatically provision dashboards.
Supports per-dashboard folder, labels, and annotations configuration.

## Prerequisites

Grafana must have the dashboard sidecar enabled and configured to watch ConfigMaps with label `grafana_dashboard: "1"`.

Example for the [Grafana community chart](https://github.com/grafana/helm-charts/tree/main/charts/grafana):

```yaml
sidecar:
  dashboards:
    enabled: true
    label: grafana_dashboard
    labelValue: "1"
    folderAnnotation: grafana_folder
```

## Bundled dashboards

| Dashboard | Grafana folder | Metrics source | Description |
|-----------|----------------|------------------|-------------|
| `temporal` | Temporal | Prometheus / VictoriaMetrics | Temporal SDK workflow and activity RED metrics |
| `microservices-otel` | Microservices | OpenTelemetry | HTTP/gRPC RED, Go runtime metrics (OTel semconv) |

## Configuration

Each dashboard entry under `dashboards` creates one ConfigMap. The sidecar discovers ConfigMaps by `sidecar.labels` and places dashboards into Grafana folders via `sidecar.folderAnnotation`.

Example — disable one dashboard and override a folder:

```yaml
dashboards:
  temporal:
    enabled: false
  microservices-otel:
    folder: "Platform"
```

### Add a new dashboard

1. Add the JSON file under `dashboards/<category>/`.
2. Register it in `values.yaml` under `dashboards.<name>`.
3. Bump `version` in `Chart.yaml` before merging to `main` (triggers a new release).
